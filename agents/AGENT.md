# [AGENTS.md](http://AGENTS.md)

本文件定义 Codex 在本仓库中的默认工作方式。目标是：准确理解需求，小步实现，充分验证，保持代码和文档可维护。

## 1. 基本原则

- 优先复用现有实现、官方 SDK 与主流生态，避免新增不必要的自研维护面。
- 先阅读相关代码、测试和文档，再开始修改。
- 复杂任务先规划，简单任务可直接实施。
- 所有结论必须基于代码、文档、命令输出或可验证事实，不得凭空猜测，禁止进行任何模糊表述。
- 保持改动小而清晰；每一步都应尽量处于可验证状态。
- 每次改动完成必须告知用户改动了哪些代码，作用是什么，这些改动会造成哪些影响。

## 2. 实施方式

- 先定位相关模块、依赖、入口、测试与相似实现。
- 明确输入输出、边界条件、错误处理与验收标准。
- 优先沿用项目既有模式、命名、构建方式和测试框架。
- 发现缺陷时，优先修复已有问题，再扩展功能。
- 对复杂变更，记录关键决策、风险和验证结果。

## 3. 编码要求

- 遵循项目既有代码风格、目录结构、命名规则和格式化配置。
- 优先写可读、直接、职责清晰的实现，避免过早抽象和炫技式写法。
- 禁止进行无底线地兜底，
- 非必要不保留兼容层；确需破坏性变更时，明确影响范围，寻求用户确认。
- 注释应说明意图、约束和使用方式；注释语言默认使用中文。
- 可以向用户请求确认，删除无用代码、过时代码和重复实现。

## 4. 验证要求

- 修改代码时，补充或更新对应测试。
- 至少执行与改动直接相关的检查：单元测试、冒烟测试、集成/功能验证、lint、类型检查、构建检查（按项目实际可用项执行）。
- 无法执行的验证需要说明原因、影响范围和剩余风险。
- 交付前确认：实现符合需求，测试通过，文档已同步，风险已说明。

## 5. 文档与留痕

- 影响使用方式、接口行为、配置方法或目录结构的改动，必须同步更新文档。
- 仅记录关键留痕：需求理解、关键决策、异常情况、失败原因、验证结果。
- 过程文件写入项目内 `.codex/`，不要写入用户主目录。

## 6. 用户确认边界

以下情况需要先确认：

- 删除核心配置或文件
- Commit 当前改动、 Push改动到远程分支
- 会造成大范围删除、或不可逆迁移或外部系统影响的操作
- 用户明确要求确认的操作

除以上情况外，默认可自主执行并在结果中说明依据与影响。

## 7. 目录级覆盖

- 若子目录存在更具体的 `AGENTS.md`，以更具体目录中的规则为准。
